Azure CI/CD Pipeline

Basic overview of Continuous Integration and Continuous Deployment using Azure DevOps

What is CI/CD?

CI (Continuous Integration) is the practice of frequently merging code changes into a shared repository and automatically building and testing the project.

CD (Continuous Delivery/Deployment) is the practice of automatically deploying the application to a target environment (such as staging or production) after it passes all checks.

Azure DevOps Pipeline Components

  • Repository (Repo): Stores your application source code.
  • Pipeline: Defines the steps to build, test, and deploy your app.
  • Agent: A machine that runs the pipeline jobs.
  • Artifacts: Build outputs that are passed to the release/deployment stage.
  • Environment: Target like Dev, QA, or Production to deploy your app.

Sample Azure Pipeline (YAML)

This is a very basic example of an Azure CI/CD pipeline for a web app:

# azure-pipelines.yml
trigger:
  - main

pool:
  vmImage: 'ubuntu-latest'

steps:
- task: UseNode@1
  inputs:
    versionSpec: '18.x'
  displayName: 'Use Node.js'

- script: |
    npm install
    npm test
    npm run build
  displayName: 'Install, Test, Build'

- task: CopyFiles@2
  inputs:
    contents: 'dist/**'
    targetFolder: '$(Build.ArtifactStagingDirectory)'
  displayName: 'Copy build files'

- task: PublishBuildArtifacts@1
  inputs:
    pathToPublish: '$(Build.ArtifactStagingDirectory)'
    artifactName: 'drop'
  displayName: 'Publish Artifacts'

Typical CI/CD Flow in Azure

  1. Developer pushes code to the main branch on Azure Repos or GitHub.
  2. The Azure pipeline is triggered automatically.
  3. The code is built and tests are executed.
  4. Build artifacts are created and published.
  5. A release pipeline (or deployment stage) uses these artifacts to deploy the app to Azure (Web App, VM, Kubernetes, etc.).
